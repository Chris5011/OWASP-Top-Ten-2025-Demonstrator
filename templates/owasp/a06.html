A06: Insecure Design — Negative Quantity Shop

Insecure Design describes vulnerabilities that exist because the application was designed without security requirements — not because a developer made a coding mistake. The code is often technically correct; it faithfully implements a design that was never thought through from a security perspective. No amount of code review or pen-testing can "patch" a fundamentally flawed design — it must be rethought.

Insecure Design vs. Coding Bug

Coding Bug

The developer intended to validate input but made an error in implementation. A code fix resolves the problem.

if quantity > 0: # developer forgot ">="
accept_order()

Insecure Design

The requirement "quantity must be positive" was never written down, never tested, and never enforced anywhere. The code does exactly what was designed — the design itself is missing a constraint.

# No validation, because nobody asked for it
total = price * quantity
accept_order(total)

Threat modeling insight:

A simple STRIDE analysis of the checkout flow would immediately surface "user enters invalid quantity" as a threat under Tampering. The mitigation — enforce positive quantity — becomes a design requirement before a line of code is written.

More Business Logic Flaw Examples

Coupon stacking

A discount code that can be applied multiple times per order. Design never specified "one coupon per order."

Race condition on limited stock

Two users check "1 item in stock" simultaneously, both add it to cart, both complete payment. Design never modeled concurrent access.

Price tampering via API

A mobile app sends the price of an item in the checkout POST request. Design trusted the client to send the correct price.

Password reset link reuse

A reset link works indefinitely, or does not invalidate after first use. Design never specified reset token expiry.

Teaching Point

Insecure Design is not just a missing if statement. The deeper problem is that the required business rule was never designed, specified, tested, or enforced at the architecture level. The fix starts before the first line of code — in requirements and threat modeling.

Bad Design

total = price * quantity
accept_order(total)

Math is correct. Business rule is missing.

Better Design

require quantity >= 1
require total >= 0
then accept_order(total)

Business constraints are explicit requirements — tested, reviewed, and enforced in code.
