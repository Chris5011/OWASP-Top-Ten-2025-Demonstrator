A07: Authentication Failures — JWT Auth Bypass

This demo reframes the JWT vulnerability from A04 (integrity) as an authentication failure: the application uses the decoded token claims to determine who the user is and what they are allowed to do — before verifying that the token has not been tampered with. The result is that an attacker who can modify the token can assume any identity or role.

A04 vs A07 — Different Lens, Same Token

A04 focus: Cryptographic Integrity

The question is: does the algorithm and signature verification prevent the token from being modified undetected? A04 covers weaknesses like accepting alg=none or skipping signature checks.

A07 focus: Authentication Flow

The question is: does the application use the token's identity claims correctly? A07 covers reading and trusting identity/role claims before verification, missing token expiry checks, and treating decoded data as authenticated facts.

Algorithm Confusion Attack

Some libraries accept multiple signature algorithms. If the server does not pin the expected algorithm, an attacker can experiment:

alg=none attack

{"alg":"none"} + modified_payload + (empty_signature)

→ Library accepts unsigned token as valid

RS256 → HS256 confusion

Server normally uses RS256 (asymmetric).
Attacker submits HS256 token signed with the
public key (which the attacker knows).

→ Confused library verifies HMAC using public key

Both attacks succeed because the application let the token dictate the verification method. The server must decide the algorithm, not the token.

5. OWASP Focus

Authentication Failure

The application reads role and identity from the JWT payload before verifying the signature. This is equivalent to asking a stranger their name and believing them before checking their ID.

decode(token) → trust(user, role) → grant access

Correct Flow

Verification happens first. Only after the server confirms the token's integrity with its own secret key are the claims accepted as representing a real user.

verify_signature(token, server_key)
→ decode(claims)
→ authenticate user
→ grant access

Teaching Point

Wrong

"JWTs are signed, so decoding them is enough."

Decoding is trivial — it is just Base64. The signature is what prevents tampering, and it must be verified before any claim is trusted.

Better

"Authentication starts after verification, not before."

Use a vetted JWT library. Configure it with your expected algorithm and secret. Let the library verify — never write signature verification manually.
