A07: Authentication Failures — Session Hijacking / Fixation

A web session represents a user's authenticated identity between requests. Because HTTP is stateless, the server issues a session token (usually stored in a cookie) that the browser sends with every request. If an attacker obtains a valid, authenticated session token, they become that user — without ever knowing the password.

Session Fixation vs. Session Hijacking

Session Fixation

The attacker plants a known session ID in the victim's browser before the victim logs in. If the application keeps the same session ID after authentication, the attacker already knows the now-authenticated ID and can use it.

1. Attacker sends victim a link with
?sessionid=KNOWN_VALUE
2. Victim logs in — same ID reused
3. Attacker uses KNOWN_VALUE → authenticated

Session Hijacking

The attacker steals an already-authenticated session ID from network traffic (via MITM on HTTP), from a browser (via XSS), or from logs. If the session is still valid, authentication is bypassed entirely.

1. Attacker intercepts / steals session cookie
2. Attacker replays cookie in own request
3. Server accepts it → authenticated

How Sessions Are Transmitted

Browser → Server (Cookie header)

GET /dashboard HTTP/1.1
Host: example.com
Cookie: session=ABCDEF123456

Recommended Cookie Flags

Set-Cookie: session=...
HttpOnly ← JS cannot read it (blocks XSS theft)
Secure ← only sent over HTTPS
SameSite=Strict ← blocks CSRF-based attacks

In safe mode, logout removes the session from the server-side store entirely. Even if an attacker had stolen the old session ID, it no longer exists — the cookie is useless. In unsafe mode, logout only clears the authenticated flag, but the session ID entry still exists and could be re-authenticated.

Teaching Point

Wrong

"The user logged in, so the session is trusted."

Authentication proves who the user is. It does not prove who holds the session token. If the token is predictable or was previously known, the login is irrelevant.

Better

On login: invalidate old session → issue new random ID
On logout: delete session server-side
Set cookies: HttpOnly + Secure + SameSite

A session rotation on login makes any previously known token useless. Server-side deletion on logout ensures the token cannot be replayed.
