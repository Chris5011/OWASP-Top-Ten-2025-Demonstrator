A04: Cryptographic Failures — JWT Token Integrity

JSON Web Tokens (JWTs) are a popular way to represent identity and claims between a client and a server. A JWT consists of three Base64URL-encoded parts: header, payload, and signature. The payload is not encrypted — anyone can read it. Security comes entirely from the server verifying the signature before trusting the claims.

JWT Structure

header.payload.signature

eyJ0eXAiOiJKV1QiLCJhbGciOiJIUzI1NiJ9 ← header: {"typ":"JWT","alg":"HS256"}
.eyJzdWIiOiJhbGljZSIsInJvbGUiOiJ1c2VyIn0 ← payload: {"sub":"alice","role":"user"}
.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw ← HMAC-SHA256 signature

The signature is computed over base64url(header) + "." + base64url(payload) using a secret key only the server knows. If an attacker changes the payload (e.g., sets role=admin), the signature no longer matches — and the server should reject the token.

Two Classic JWT Attacks

Attack 1 — alg=none

The JWT header includes an alg field that names the signature algorithm. In 2015 a widespread vulnerability was discovered (CVE-2015-9235): if the library trusted this field, an attacker could set "alg":"none", remove the signature, and submit a token the library accepted as valid — because "no signature" is treated as a valid algorithm variant.

Header: {"alg":"none"}
Payload: {"role":"admin"}
Signature: (empty)

→ Many libraries accepted this

Attack 2 — Modified payload, old signature

An attacker edits the payload (changing role to admin) but keeps the original signature from before the edit. A naive implementation that does not recompute the expected signature will fail to detect the mismatch.

Original: header.original_payload.valid_sig
Tampered: header.admin_payload.valid_sig

→ Signature no longer matches payload

Key principle

A JWT is not a proof of identity — it is a claim about identity. The claim only becomes trustworthy after the server independently verifies the signature using a secret key the attacker does not have. The server must never trust the token's own alg field.

In safe mode, the server recomputes the expected HMAC-SHA256 signature over the current header and payload using its own secret key, then compares it against this value using a constant-time comparison (to prevent timing attacks). Any mismatch — including an empty signature from alg=none — causes rejection.

Teaching Point

Wrong

"The token says admin, so the user is admin."

The payload is attacker-controlled. Reading claims before verifying the signature is the vulnerability.

Better

1. Enforce expected algorithm server-side
2. Verify signature with your secret key
3. Only then read and trust the claims

Use a well-maintained JWT library. Never implement signature verification manually. Pin the expected algorithm in code — never derive it from the token.
