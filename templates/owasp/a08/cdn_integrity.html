A08: Software or Data Integrity Failures — CDN Tampering and SRI

When a web page loads JavaScript from a third-party CDN, that script runs in the user's browser with the same permissions as your own code — it can read the DOM, steal cookies, make authenticated API calls, and exfiltrate data. If the CDN is compromised, or if the URL resolves to different content than expected, every visitor to your site executes the attacker's code — silently.

Subresource Integrity (SRI)

Subresource Integrity is a browser security feature that lets you bind a specific expected hash to an external script or stylesheet. The browser computes the hash of the downloaded file and compares it to the value in your HTML. If they do not match, the browser refuses to execute the script.

Without SRI

<script src="https://cdn.example.com/lib.js"></script>

The browser downloads whatever is at that URL and runs it. If the CDN is compromised tomorrow, your users run malicious code tomorrow.

With SRI

<script
src="https://cdn.example.com/lib.js"
integrity="sha384-abc123..."
crossorigin="anonymous"></script>

The browser computes the SHA-384 hash of the downloaded file and checks it against abc123.... A modified script produces a different hash — the browser blocks it before it executes.

How to Generate an SRI Hash

# Using openssl:
curl -s https://cdn.example.com/lib.js | openssl dgst -sha384 -binary | openssl base64 -A

# Using the SRI Hash Generator website:
https://www.srihash.org/

# Result goes in the integrity attribute:
integrity="sha384-<your_hash_here>"

The hash must be recomputed every time the file legitimately changes (e.g., a library version upgrade). This is intentional — it forces a conscious review before trusting new third-party code.

Supply Chain Attack Context

CDN compromise is one vector in a broader class of software supply chain attacks:

CDN compromise

Attacker gains write access to a CDN bucket or DNS and replaces a legitimate file with a malicious version. All sites linking to it are instantly affected. Notable: Polyfill.io (2024) — the domain was acquired and began serving malware to millions of sites.

npm package hijack

Attacker publishes a malicious version of a popular npm package, or takes over a maintainer account. Any project that runs npm install pulls down the malicious code. Notable: event-stream (2018) — malicious code targeting cryptocurrency wallets was embedded in a widely-used package.

Teaching Point

Wrong

"It comes from a CDN, so it is probably fine."

CDNs are targets. Libraries are abandoned and taken over. Trusting a URL is trusting every future version of whatever is served at that URL.

Better

"Pin exact versions. Add integrity hashes.
Review changes before bumping versions."

SRI gives the browser the ability to enforce integrity. Self-hosting critical scripts removes the CDN as a trust dependency entirely.
