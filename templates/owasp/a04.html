A04: Cryptographic Failures — Password Hashing

Cryptographic failures occur when sensitive data is protected with the wrong tool — or with no protection at all. This module focuses on password storage. Many developers reach for "hashing" without understanding that not all hash functions are designed for passwords. The wrong choice leaves users vulnerable even after a database breach.

Why Speed Is the Enemy

When an attacker gets a copy of your database, the game becomes offline brute-forcing: they hash billions of guesses per second and compare results against your stored hashes.

AlgorithmHashes per second (GPU)Time to crack 8-char password
MD5~60 billionMinutes to hours
SHA-256~12 billionHours to days
bcrypt (cost=12)~150 thousandYears
Argon2id (tuned)~1 thousandCenturies

Password hashing algorithms like Argon2id are deliberately slow and memory-intensive. Their work factor can be tuned upward as hardware gets faster — giving defenders a permanent advantage over offline attackers.

5. Comparison Cheat Sheet

Method Good For Password Storage? Main Problem / Note
Plaintext Nothing security-related ❌ Never Immediate exposure after database leak
SHA-256 Integrity checks, checksums ❌ No Too fast; ~12 billion hashes/sec on a modern GPU
Salted SHA-256 Explaining salts ⚠️ Still no Salt prevents rainbow tables but hashing speed is still the problem
HMAC-SHA-256 API signatures, token integrity ⚠️ Not by itself Keyed MAC primitive — not designed for password work factor
bcrypt Password storage (legacy) ✅ Yes Limited to 72-byte passwords; prefer Argon2id for new systems
Argon2id Password storage ✅ Yes Memory-hard + CPU-hard; tunable; NIST recommended

Teaching Point

"Hashing" is not a single thing. SHA-256 and Argon2id are both hash functions, but they were designed for completely different purposes. The question to ask is: what threat model does this primitive protect against?

Wrong

"It is hashed, so it is safe."

SHA-256 is fast by design — that property is a vulnerability when the hash is a password.

Better

"Use Argon2id with tuned parameters.
Let the library handle salt generation."

Argon2id is slow and memory-intensive by design. Tuning ensures it remains expensive as hardware improves.
