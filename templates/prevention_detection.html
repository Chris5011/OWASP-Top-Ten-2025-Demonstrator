Prevention & Detection

The general pattern behind injection issues is that untrusted input crosses a boundary from data into instructions. Fixes should therefore enforce this separation.

1. Prevent Injection

SQL / LDAP / NoSQL

Use parameterized queries, prepared statements, query builders, or ORM defaults that bind values separately from query logic.

query = "SELECT * FROM users WHERE username = ?"
bind(username)

XSS

Use contextual output encoding. Encode data differently depending on whether it appears in HTML text, attributes, JavaScript, CSS, or URLs.

<div>{{ escaped_user_input }}</div>

Command Injection

Avoid shell execution where possible. Prefer safe APIs, argument arrays, allow-listed input, and fixed command structures.

subprocess.run(["ls", "-l", safe_filename], shell=False)

2. Defense-in-Depth

CSP for XSS

Content Security Policy can limit the impact of XSS by restricting script sources and blocking inline script execution.

Content-Security-Policy: default-src 'self'; script-src 'self'

Least Privilege

Databases, web servers, service users, and API tokens should only have the permissions they actually need.

www-data should not be able to modify system files

Normalize / Reject Ambiguity

For HTTP request smuggling, reject ambiguous requests and ensure front-end and back-end components agree on HTTP parsing and framing.

3. Detection Techniques

SAST

Static Application Security Testing checks source code for risky patterns before the application is running.

Semgrep: custom rules for unsafe string concatenation
SonarQube: code quality and security issue detection
Look for: raw SQL strings, shell=True, unsafe HTML rendering

DAST

Dynamic Application Security Testing tests the running application from the outside.

OWASP ZAP active scan
Fuzz input fields and parameters
Watch for changed responses, errors, or unexpected behavior

4. Quick Mapping

Injection Type Primary Fix Detection Ideas
SQL Injection Prepared statements, ORM parameter binding SAST for string-built queries; DAST with ZAP
Command Injection No shell, argument arrays, allow-listed input SAST for shell execution; logging unexpected metacharacters
XSS Contextual output encoding SAST for unsafe sinks; DAST browser-based checks
LDAP Injection LDAP filter escaping / safe directory APIs SAST for raw filter construction
NoSQL Injection Strict schema validation, disallow query operators in user input SAST for user-controlled query objects
HTTP Request Smuggling Reject ambiguous framing; align proxy/backend parsing DAST/proxy tests; log malformed transfer headers

5. Teaching Summary

The safest model is: never concatenate untrusted input into an interpreter language. Bind, encode, validate, normalize, and restrict privileges.
