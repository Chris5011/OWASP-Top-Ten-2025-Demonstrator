HTTP Request Smuggling: Parser Disagreement Example

HTTP request smuggling exploits a discrepancy between how a front-end proxy and a back-end server determine where one HTTP request ends and the next begins. By crafting a request that the two components parse differently, an attacker can "smuggle" a hidden second request into the pipeline — bypassing access controls, poisoning caches, or hijacking other users' requests.

The Two Framing Headers

HTTP/1.1 offers two ways to specify where a request body ends:

Content-Length (CL)

Declares the exact number of bytes in the request body. The receiver reads exactly that many bytes and treats the rest as the start of the next request.

Content-Length: 13

[13 bytes of body]

Transfer-Encoding: chunked (TE)

The body is sent in chunks, each preceded by its length in hexadecimal. A chunk of size 0 signals the end of the body.

Transfer-Encoding: chunked

5\r\n
Hello\r\n
0\r\n
\r\n

CL.TE: The Classic Smuggling Pattern

When both headers are present in the same request, RFC 7230 says Transfer-Encoding takes priority and Content-Length should be ignored. But many real-world stacks do not implement this uniformly.

Front-end proxy uses Content-Length

The proxy reads Content-Length: 13 bytes, thinks the request is complete, and forwards it to the back-end.

Back-end server uses Transfer-Encoding

The server reads chunked encoding. After the 0\r\n\r\n terminator it finds extra bytes — which it treats as the start of a new request.

Those leftover bytes are the attacker's "smuggled" request. Because they appear to come from the front-end proxy rather than from the attacker's connection, they may bypass IP-based access controls or appear to the back-end as a request from a trusted internal source.

Real-World Impact

What an attacker can achieve

  • Bypass front-end access controls (WAF, authentication proxy)
  • Poison the back-end connection queue to hijack other users' requests
  • Steal session tokens or credentials from other users' requests
  • Achieve reflected XSS on sites that are otherwise CSP-protected
  • Poison the reverse proxy cache to serve attacker content to all users

Correct fix

  • Reject any request that includes both Content-Length and Transfer-Encoding
  • Ensure the front-end and back-end use the same HTTP parsing library and version
  • Prefer HTTP/2 end-to-end — it has a single unambiguous length mechanism
  • Use modern reverse proxies (nginx ≥1.21, HAProxy ≥2.2) with strict CL/TE handling

Teaching Point

Request smuggling is subtly different from the other injection demos. The attacker does not inject into a query language or shell — they exploit a disagreement between two components about the protocol boundary. The root cause is ambiguity in the HTTP spec combined with non-uniform implementations. The fix is to eliminate the ambiguity entirely rather than to sanitize input.
