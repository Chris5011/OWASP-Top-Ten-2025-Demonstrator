XML Injection: Profile Update Example

XML is a structured data format widely used in configuration files, SOAP web services, document storage, and data interchange. When an application builds an XML document by concatenating user input without escaping, the attacker can inject markup that alters the document structure — breaking parsing, adding new elements, or overriding existing ones.

How XML Structure Injection Works

Imagine an application stores a user's display name in XML:

<user><name>Alice</name><role>student</role></user>

Attacker submits:

Alice</name><role>admin</role><name>

Document becomes:

<user><name>Alice</name><role>admin</role><name></name><role>student</role></user>

Depending on how the application reads back the XML, the extra <role>admin</role> element may take precedence — effectively granting the user admin privileges.

With escaping:

Alice&lt;/name&gt;&lt;role&gt;admin&lt;/role&gt;&lt;name&gt;

Document stays intact:

<user><name>Alice&lt;/name&gt;...</name><role>student</role></user>

Escaped characters appear as visible text inside the element — they are never parsed as markup.

Related: XML External Entity (XXE) Injection

A more severe sibling of XML injection is XXE. XML parsers can be configured to resolve external entity references — URIs that point to local files or internal network addresses. If user-supplied XML is parsed with external entities enabled, an attacker can read files like /etc/passwd, probe internal services, or in some parsers trigger denial of service via a "billion laughs" entity expansion.

<?xml version="1.0"?> <!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]> <user><name>&xxe;</name></user>

Fix: disable DTD processing and external entity resolution in your XML parser. In Java use FEATURE_SECURE_PROCESSING; in Python use defusedxml.

Real-World Impact

What an attacker can achieve

  • Inject additional XML elements that override role or permission fields
  • Corrupt stored XML data or configuration files
  • Exploit XXE to read local files or probe internal network services
  • Trigger denial of service via recursive entity expansion

Correct fix

  • Build XML using DOM APIs (e.g., xml.etree.ElementTree) rather than string concatenation
  • If concatenating, escape < > & ' " before insertion
  • Disable external entities and DTD processing in your XML parser
  • Use JSON or other simpler formats when XML is not required

Teaching Point

XML injection follows the same pattern: user input crosses from character data into XML markup. Escaping makes that crossing structurally impossible — the characters that have meaning in XML syntax are encoded so the parser treats them as text, not as instructions.

Unsafe

"<name>" + user_input + "</name>"

Safe

element = ET.SubElement(user, "name")
element.text = user_input
