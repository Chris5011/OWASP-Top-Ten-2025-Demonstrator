NoSQL Injection: MongoDB-style Login Example

NoSQL databases like MongoDB do not use SQL, but they still have a query language with operators and logic. In MongoDB, queries are sent as JSON objects. If an application constructs a query by embedding user-provided strings without strict typing, an attacker can inject JSON operators that change the query's meaning — the NoSQL equivalent of SQL injection.

How MongoDB Query Injection Works

A normal login query checks that both fields match exactly:

db.users.findOne({ "username": "alice", "password": "secret" })

Password field injection: { "$ne": null }

db.users.findOne({
"username": "admin",
"password": { "$ne": null }
})

$ne means "not equal". { "$ne": null } matches any document where the password field is not null — which is every real user account. No password needed.

Username injection: { "$where": "true" }

db.users.findOne({
"username": { "$where": "true" },
"password": "anything"
})

$where evaluates a JavaScript expression server-side. true matches every document. In older MongoDB versions this could also be used for arbitrary JavaScript execution.

Safe: strict string typing

username = str(request.form["username"]) # guaranteed to be a string
db.users.findOne({ "username": username, "password": password })

When the application enforces that both fields are plain strings, the driver sends them as JSON strings — never as objects containing $ne, $gt, or $where. The operator injection becomes impossible.

Common Vulnerable Query Operators

OperatorMeaningInjection Effect
$neNot equal{ "$ne": null } matches any non-null value — bypasses exact match
$gtGreater than{ "$gt": "" } matches any non-empty string password
$regexRegular expression matchCan be used to brute-force a value character by character
$whereJavaScript expressionArbitrary server-side JS execution in older MongoDB versions

Real-World Impact

What an attacker can achieve

  • Bypass authentication entirely without knowing any password
  • Enumerate documents by using comparison operators ($gt, $regex)
  • Execute arbitrary server-side JavaScript in vulnerable MongoDB configurations
  • Extract sensitive data from other collections if the query scope is not limited

Correct fix

  • Enforce strict types: cast input to string before using in a query
  • Use an ODM (e.g., Mongoose with schema validation) that rejects unexpected operator objects
  • Disable server-side JavaScript evaluation (security.javascriptEnabled: false)
  • Validate input shape before it reaches the database layer

Teaching Point

NoSQL injection is not SQL injection — but it is the same underlying problem. The query language changed; the risk did not. Whenever user input can influence the structure of a query — whether SQL, LDAP, JSON, or XML — injection is possible if that boundary is not enforced.
