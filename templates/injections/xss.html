Cross-Site Scripting (XSS): Comment Example

Cross-Site Scripting (XSS) occurs when an application includes user-supplied content in a web page without properly encoding it. The browser receives what looks like HTML or JavaScript from the trusted server and executes it — not knowing the code came from an attacker. XSS is consistently one of the most widespread web vulnerabilities.

Three Types of XSS

Reflected XSS

The malicious script is embedded in a URL parameter. The server reflects it back in the response without sanitization. The attacker tricks a victim into clicking a crafted link.

https://example.com/search?q=<script>steal()</script>

Stored XSS

The script is saved to the database (e.g., in a comment or profile field) and later rendered to every visitor who views that content. This is the most dangerous type — this demo simulates stored XSS.

Comment stored: <script>steal()</script>
Rendered to all visitors

DOM-Based XSS

The vulnerability lives entirely in JavaScript. The page reads a value from a source (e.g., location.hash, document.cookie) and writes it into the DOM via a sink (e.g., innerHTML) without encoding. The server is never involved.

document.getElementById("msg").innerHTML = location.hash.slice(1);

Real-World Impact

What an attacker can achieve

  • Steal session cookies and hijack authenticated sessions
  • Log keystrokes and capture passwords as the victim types
  • Redirect the victim to a phishing page
  • Make requests to the application on behalf of the victim (CSRF-like)
  • Exfiltrate sensitive data visible on the current page

Correct fix

  • Use contextual output encoding — HTML, JS, CSS, and URL contexts need different escaping
  • Use templating engines that auto-escape by default (Jinja2, Twig, Razor)
  • Add a Content Security Policy header to limit script sources
  • Set cookies HttpOnly to prevent JavaScript from reading them

Teaching Point

XSS follows the same injection pattern: untrusted input crosses the boundary from text/data into browser instructions. The interpreter here is the browser's HTML/JS engine rather than a SQL engine or shell.
