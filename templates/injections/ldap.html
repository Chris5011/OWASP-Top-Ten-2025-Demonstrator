LDAP Injection: Directory Login Example

LDAP (Lightweight Directory Access Protocol) is the protocol used by enterprise directory services such as Microsoft Active Directory and OpenLDAP. Applications often query a directory to authenticate users or look up their roles and group memberships. LDAP injection occurs when user input is inserted into an LDAP filter without escaping, allowing an attacker to alter the filter's logic — just as SQL injection alters a SQL query.

How LDAP Filters Work

An LDAP filter uses a prefix-notation syntax. The filter below means "find entries where uid equals alice AND userPassword equals secret":

(&(uid=alice)(userPassword=secret))

Wildcard attack — Username: *

(&(uid=*)(userPassword=anything))

The asterisk is the LDAP wildcard. uid=* matches every entry in the directory. The first matching user — often the admin — is returned.

Filter escape — Username: admin)(|(uid=*)

(&(uid=admin)(|(uid=*))(userPassword=anything))

The injected parentheses close the AND clause and open a new OR clause. Because uid=* is always true, the entire filter evaluates to true and returns all users.

Real-World Impact

What an attacker can achieve

  • Bypass authentication without knowing any password
  • Enumerate all usernames in the directory
  • Retrieve sensitive attributes (email, phone, group memberships)
  • Potentially access the full Active Directory schema

Correct fix

  • Escape all user input per RFC 4515 before inserting into a filter
  • Prefer high-level directory SDK methods that parameterize filters
  • Validate that usernames match an expected pattern (letters, digits, hyphen) before the LDAP call
  • Apply least privilege: the bind account should be read-only

Teaching Point

LDAP injection is the same root cause as SQL injection, applied to a different interpreter. User input crosses from directory attribute value into LDAP filter logic.

Unsafe

(&(uid=USER_INPUT)(userPassword=USER_INPUT))

Safe

(&(uid=escape(USER_INPUT))(userPassword=escape(USER_INPUT)))
